DMR Radio Basics Lesson 12

Data & Text Messaging

DMR carries more than voice — text, GPS, presence, control commands. The rule that governs all of it: on a given slot, at a given instant, it's voice or data, never both.

Beyond the voice you've studied, DMR moves data: SMS-style text, GPS locations, registration, and short control commands. For your bridge these are candidate PoC features (chat, map pins, presence). But two things shape what's realistic: data is slow, and it competes with voice for the slot.

The layering matters: ETSI TS 102 361-3 defines the data protocol (confirmed/unconfirmed transport, packet data, retries); the air-interface burst/header formats are in -1. But TMS, ARS, and LRRP are application layers vendors built on top — not ETSI. "It's DMR text" really means "Motorola's protocol over DMR's data service."1

How a text message actually goes out

A text is not voice — it's a run of data bursts on one time slot, addressed to a Radio ID or talkgroup:2

PreambleCSBK, wake/aim
→
Data headerconfirmed? size? dest
→
Data blocksRate ½ / ¾ ×N
→
Last block+ ACK if confirmed

Confirmed delivery uses link-layer ACKs and selective retry — the receiver says which blocks it missed and only those are re-sent. Unconfirmed is fire-and-forget. Longer messages fragment across multiple blocks, reassembled via the header's block count and per-block CRCs.3

The big rule: voice OR data, never both on a slot

DMR is 2-slot TDMA on a 30 ms cycle. Any single burst is either a voice burst or a data/control burst — different burst types sharing the same slot structure, never simultaneous.2 A receiver tells them apart by the 48-bit sync pattern (distinct values for voice vs data, MS- vs BS-sourced — the same mechanism from Lesson 8).

TS1 — voice call in progressdata for these users must wait…
TS2 — free → a text can go hereor queue until TS1 frees

Voice has priority; data queues behind it — or rides the other slot if free. (Motorola's "data over voice interrupt" can preempt, but it's a proprietary extra, not standard, and native text/GPS don't use it.)4

How fast? Slow.

A slot yields only a few kbps after FEC — a Rate ¾ data call is ~2–3 kbps usable; a GPS packet is under ~600 ms on air. Text and short GPS/status are the realistic payloads; bulk/file transfer is impractical and steals slot time from voice.4

What this means for your bridge
  • Natural PoC mappings: TMS text ↔ PoC chat (preserve Radio ID/talkgroup addressing and confirmed-vs-unconfirmed = delivery receipts); LRRP GPS ↔ PoC map pins; ARS ↔ PoC presence/online.
  • SDK reality check: whether you get TMS/LRRP/ARS as parsed events depends on the vendor SDK, not the chip. Some expose only raw short/packet-data send-receive and leave you to implement MBXML and the Motorola/Hytera app layers yourself (open libs like ok-dmrlib are the reference).
  • Plan for latency: because voice always preempts data, message delivery is best-effort with variable latency — don't promise instant text while a talkgroup is busy.
  • Keep it small: design the bridge around text + location, not data pipes; DMR's data rate won't carry more.
