DMR Radio Basics Lesson 10

Stun, Kill & Revive

Remotely disabling a radio. The feature is real and widely used — but where it lives in the stack (spec? firmware? app?) is the part that decides whether your bridge can ever touch it.

You asked the sharp question directly: is this in the spec, on the module, or application-side? The honest answer is all three layers at once, and which matters depends on the DMR tier. Let's separate them cleanly, because product claims and security posture both hinge on getting this right.

What the three commands mean

CommandEffectRecovery
Stun (Radio Disable / Inhibit)Radio stops normal TX/RX, won't power up normally; often still trackable/monitorableRevivable over the air
Kill (Permanent Disable)Disables everything except power on/off — for confirmed-stolen radiosTerminal — depot / CPS reprogram, not OTA
Revive (Radio Enable / Uninhibit)Re-enables a stunned radion/a — it is the recovery

The key relationship: revive undoes a stun, never a kill. Vendors name these inconsistently — Motorola says "Radio Disable / Enable" for the recoverable pair and reserves true "Kill" for its Capacity Max trunked tier; Hytera markets the full "Stun / Revive / Kill" trio.1

Layer 1 — Is it in the ETSI standard?

This is the distinction to burn in, because it splits by tier:

Standard vs vendor-specific
  • Tier III (trunked) — standardized. ETSI TS 102 361-4 defines explicit MS Stun / Revive / Kill procedures, each with and without authentication.2
  • Conventional / Tier II (the common case) — manufacturer-specific. The "Radio Disable/Enable" your MOTOTRBO or Hytera radios use rides as a manufacturer CSBK under a Feature ID (FID), not a standardized PDU.3

That FID byte is why interoperability isn't guaranteed: FID 0 is standard/DMRA-common, 0x10 is Motorola, 0x68/0x08 is Hytera. A Motorola-format stun will generally not disable a Hytera radio on a conventional system, and vice-versa.3 "Stun/kill is part of DMR" is only true for Tier III — say so precisely in any product claim.

Layer 2 — Enforced on the module (firmware)

Whatever the origin, the target radio disables itself. The command is just an individually-addressed message; the radio's own firmware is what acts on it. Critically, this is gated in the codeplug: a "Radio Disable Decode" option can be turned off, after which the radio ignores all disable commands.4 No decode support → the command does nothing. So enforcement is entirely a firmware/config property of the target.

Layer 3 — Originated by an application (or a supervisor radio)

The sender is typically a dispatch / console application (RDAC, SmartPTT, TRBOnet, via a control station) or another radio whose keypad signalling is enabled — Motorola recommends restricting that to supervisors. True Capacity-Max Kill can only come from a dispatch application, never radio-to-radio.4

How it travels — and why it's weakly protected

Originator (dispatch / supervisor radio)
Target radio (firmware enforces)
Knows the target's individual Radio ID, color code, (and any privacy key)
Stun command — CSBK addressed to the Radio ID auth optional
Acknowledgement (recoverable path ACKs success)
Target now disabled — may still register / be tracked / await revive
Revive command — re-enables the stunned radio
Acknowledgement

It's a short, individually-addressed CSBK — not voice. The target must be powered on, in range, and on the right channel to hear it; a plain conventional disable is not reliably queued if the radio is off.4

Security reality Conventional Radio Disable is effectively spoofable: anyone who knows the radio's ID, frequencies, color code and any privacy key can disable (or re-enable) it — there's no mandatory crypto by default. And ETSI's Tier III "without authentication" variants mean even trunked stun/kill may run unauthenticated depending on config.1
What this means for your bridge
  • Can your bridge ORIGINATE stun/kill/revive? Only if both hold: (1) the DMR module's SDK actually exposes an API to send these PDUs — often it doesn't, as it's a privileged supervisor/dispatch feature — and (2) the target fleet honors that exact command (same vendor/FID with decode enabled, or the standardized Tier III path). Treat it as "probably vendor-locked / unavailable" until the module's SDK docs prove otherwise.
  • Can your own device be stunned/killed? Only if its DMR module decodes disable for its Radio ID. If you disable Radio Disable decode in the module's config, the DMR side is immune. A stun would knock out the DMR link but not the Android/PoC side.
  • It's a DoS surface both ways. A bridge that can originate disables is a powerful denial tool; one that exposes its module's Radio ID is a denial target. Protect Radio IDs; prefer authenticated mechanisms; default decode off unless you truly need OTA management.
  • Mapping to PoC: there's no native PoC equivalent unless you build one — decide whether "disable a unit" is a feature your platform offers, and never silently bridge a spoofable command into a privileged PoC action.
