DMR Radio Basics Lesson 14

Standard vs Vendor Features

Which DMR features work across any compliant radio, and which only inside one vendor's fleet. One byte — the Feature ID — is the lens that answers it.

You've now met most DMR features across the course. The product-critical question that cuts across all of them: if our radios talk to a customer's existing fleet of a different brand, what's guaranteed to work? The answer isn't feature-by-feature folklore — it's governed by where each feature is specified.

The lens: the Feature ID (FID)

Every control message (CSBK) and many PDUs carry a FID byte saying which feature set defines its meaning:1

ETSI in TS 102 361 — interoperable by definition DMRA / FID 0 common air-interface service — candidate interop Vendor FID 0x10 Motorola, 0x08/0x68 Hytera — not guaranteed cross-vendor App layer protocol over the data service — often cloned, verify Local codeplug only — never crosses the air

FID 0x00 → any compliant radio can interpret it. FID 0x10 or 0x68 → the payload means only what that vendor says. That single byte is your best predictor of "will it interoperate."1

The two piles that matter

Safe across any compliant DMR radio The air interface itself (TDMA, color code, slots, LC, CACH), group/private/all call, clear voice, direct mode/talkaround, the data bearer, the raw emergency flag + priority, and Tier III trunking. If it's the bedrock of "being a DMR radio," it's standard.
Assume single-vendor only — verify before promising Encryption (the highest-risk item — "standard on paper, fragile in practice"), stun/remote monitor, Man Down / Lone Worker / hot mic / revert / silent, vendor trunking (Cap Plus/Max, Connect Plus, XPT), vendor text/GPS/ARS, and IP Site Connect. These ride vendor FIDs or app layers and are not guaranteed across brands.
What this means for your bridge
  • Build the core on the standard pile. Voice + group/private call + the emergency flag are interoperable — anchor your DMR↔PoC routing there and it works against any compliant fleet.
  • Treat the vendor pile as "configurable, per-deployment." If a customer wants stun, encryption, or LRRP GPS bridged, that's a vendor-specific integration scoped to their radios — not a universal feature. Don't promise it generically.
  • Encryption is a contract, not a checkbox. Same name ≠ same bytes. Never claim cross-vendor encrypted interop without bench testing the exact radios.
  • Your own module's FID matters: what your DMR module emits/decodes determines which fleets you slot into. Confirm its FID support against each target fleet.
